# AI-First Product Owner — Take-Home Assessment

## Overview

This is an async, spec-driven product exercise designed to evaluate how you think, design, and build using AI-first workflows. You will identify a problem worth solving, define requirements, design a system, and implement a working application using Kiro.

**Time expectation:** 3–5 hours (not timed — complete at your own pace within 5 business days)
**Deliverable:** A public GitHub repository containing your complete project

---

## The Challenge

Build something useful in the finance or fintech space — for example, loan origination, payments processing, account onboarding, budgeting, credit decisioning, or another idea or tool that showcases your experience and capabilities.

You decide what to build, who it's for, and why it matters. Pick something that lets you demonstrate product thinking, not just technical ability.

### Constraints

1. **Domain:** Your application must be relevant to finance, banking, or fintech. It should solve a real problem a real user might have.

2. **External API integrations:** If it makes sense for your build, integrate one or more public APIs. The choice of API should serve your product — pick whatever supports the experience you're building.

3. **Built with Kiro:** The project must be built using Kiro's spec-driven workflow. Other AI tools (ChatGPT, Claude, Copilot, etc.) can be used as supplementary aids for research, brainstorming, or debugging — but Kiro should be your primary development environment and the spec workflow should drive your implementation. If you haven't used Kiro before, that's fine — figuring it out is part of the exercise.

4. **Published and runnable:** Your completed project must be in a public GitHub repository. Ideally, host a live demo (Vercel, Netlify, Railway, etc.) — but at minimum, a user should be able to clone the repo and run it locally with clear setup instructions.

---

## What You Must Deliver

Your GitHub repository must contain:

### 1. Kiro Spec Artifacts

Your project should include spec artifacts generated through Kiro's spec workflow — each containing requirements, design, and tasks. How you organize your specs — whether as a single feature or multiple increments — is a product decision.

We will review the spec quality as heavily as the code. These should reflect your product thinking — not just a restatement of this brief.

### 2. AI Collaboration Context

We need to understand how you worked with AI throughout this exercise. We want visibility into your process — the decisions you made, how you iterated, and how AI shaped (or didn't shape) the final product.

How you capture and present that context is up to you. The approach you choose is itself part of what we're evaluating.

### 3. A Working Application

It must run. Technology stack is your choice. For anything with a front-end, design and user experience matter — we're evaluating product thinking, and that includes how the experience looks and feels, not just whether it functions.

### 4. A README.md

- What you built and why — what problem does it solve? Who is the user?
- Which APIs you integrated and how they serve the product
- How to run it locally (and link to live demo if hosted)
- Product decisions you made and your reasoning
- What you would change or add with more time
